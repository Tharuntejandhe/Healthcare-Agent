# Deploying MediHealth

Production topology (no Azure required):

```
            ┌─────────────────────┐        ┌──────────────────────────┐
  Browser → │  Vercel (frontend)  │  HTTPS │  Render (FastAPI backend) │
            │  Next.js 16         │ ─────▶ │  Docker, gunicorn workers │
            └─────────────────────┘        └────────────┬─────────────┘
                                                         │ DATABASE_URL (TLS)
                                                ┌────────▼─────────┐
                                                │  Neon Postgres   │
                                                └──────────────────┘
```

- **Frontend** → Vercel (auto-deploys from GitHub).
- **Backend** → Render (Docker web service, defined by `render.yaml`; auto-deploys from GitHub).
- **Database** → Neon (serverless Postgres).
- **File storage** → local disk on the backend by default (mount a Render disk to persist). Azure Blob is optional (`STORAGE_BACKEND=azure`).

> ⚠️ **Rotate your Groq key first.** A real `GROQ_API_KEY` was present in `backend/.env` during development. Rotate it in the Groq console and only put the new value into Render's secret env vars — never commit it.

## 1. Database — Neon

1. Create a project at [neon.tech](https://neon.tech).
2. Copy the **pooled** connection string. It looks like:
   `postgresql://USER:PASSWORD@ep-xxx-pooler.REGION.aws.neon.tech/neondb?sslmode=require`
3. Keep it handy — it becomes `DATABASE_URL` on Render. (Migrations run automatically on each backend deploy.)

## 2. Backend — Render

**Option A — Blueprint (recommended).** Render reads `render.yaml`:

1. Push this repo to GitHub.
2. Render dashboard → **New + → Blueprint** → select the repo.
3. Render creates the `medihealth-backend` web service. Fill in the secret env vars it prompts for:

   | Variable | Value |
   |---|---|
   | `DATABASE_URL` | your Neon pooled connection string (`…?sslmode=require`) |
   | `BACKEND_CORS_ORIGINS` | your Vercel URL, e.g. `https://your-app.vercel.app` (comma-separate multiples) |
   | `GROQ_API_KEY` | your **rotated** Groq key |
   | `TAVILY_API_KEY` | optional (web search); leave blank to disable |
   | `GOOGLE_CLIENT_ID` | optional (Google sign-in) |
   | `SECRET_KEY` | auto-generated by Render |

4. Deploy. Render builds the Dockerfile, runs `alembic upgrade head` on boot, and serves on `$PORT`. Health check: `GET /livez`.

**Memory note:** local RAG embeddings (fastembed) need ~1 GB RAM → use the **Standard** plan. On the 512 MB free plan, set `ENABLE_RAG=false` (chat / vision / speech still work; document-grounded RAG is disabled gracefully).

**Persistence:** the blueprint mounts a 1 GB disk at `/app/data` for uploads + FAISS + the model cache. Remove the `disk:` block if you don't need persistence (files become ephemeral).

Your backend URL will look like `https://medihealth-backend.onrender.com`.

## 3. Frontend — Vercel

1. Vercel dashboard → **Add New → Project** → import the repo.
2. Set **Root Directory** = `frontend`. Framework auto-detects as Next.js.
3. Add environment variables:
   - `NEXT_PUBLIC_API_URL` = your Render backend URL (e.g. `https://medihealth-backend.onrender.com`)
   - `NEXT_PUBLIC_GOOGLE_CLIENT_ID` = your Google Web Client ID (optional)
4. Deploy.
5. **Back on Render**, set `BACKEND_CORS_ORIGINS` to the Vercel domain and redeploy (or it was already set in step 2).

## 4. Google sign-in (optional)

In Google Cloud Console → Credentials → your OAuth Web Client, add the Vercel domain to **Authorized JavaScript origins** (e.g. `https://your-app.vercel.app`). The same client ID goes into both `GOOGLE_CLIENT_ID` (Render) and `NEXT_PUBLIC_GOOGLE_CLIENT_ID` (Vercel).

## Local development

**Docker (full stack incl. Postgres):**
```bash
cp .env.example .env        # fill GROQ_API_KEY at minimum
docker compose up --build   # frontend :3000, backend :8000, postgres :5432
```

**Manual:**
```bash
# backend
cd backend && python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env        # fill in keys
alembic upgrade head
uvicorn app.main:app --reload

# frontend (new terminal)
cd frontend && npm install
cp .env.example .env.local  # set NEXT_PUBLIC_API_URL=http://localhost:8000
npm run dev
```

## What "ultra robust" means here

- **Non-blocking AI**: every LLM / FAISS / file call runs in a worker thread, so one slow request can't freeze the server for everyone.
- **Timeouts + retries** on all Groq calls; **rate limiting** on auth (`10/min`) and AI endpoints (`20/min`).
- **Upload guards**: streamed size caps (PDF 15 MB / image 10 MB / audio 25 MB) and magic-byte content validation.
- **Safe errors**: clients get a consistent `{ "error": { code, message, request_id } }` envelope; internals/PII never leak; structured JSON logs with a per-request ID.
- **Real RAG**: random mock embeddings were replaced with local `fastembed` (medically meaningful retrieval).
- **Prod safety**: the app refuses to boot in production with a weak `SECRET_KEY`, `*` CORS, or SQLite; runs as a non-root container with multiple gunicorn workers and a real `/readyz`/`/livez`.
